# TASK: Реализация нескольких Master-нод с дублированием и балансировкой

## Цель
Реализовать возможность запуска нескольких мастер-нод, которые синхронизируют состояние между собой (Raft) и доступны по адресу `http://master.local` с автоматической балансировкой.

## Задачи
1. **Создание `crates/r4a-store`**:
   - Интеграция `openraft` для консенсуса.
   - Использование `sled` для персистентного хранения.
   - Синхронизация списка пиров (VPN) и git-репозиториев (метаданных).
2. **Обновление `r4a-server`**:
   - Поддержка ролей: не просто `init`, а возможность присоединить новый мастер к существующему.
   - Каждый мастер имеет свой уникальный VPN IP (10.42.0.1, 10.42.0.2, ...).
   - Мастера соединяются друг с другом в Mesh.
3. **Балансировка `master.local`**:
   - Обновление `r4a-agent`, чтобы он прописывал все IP мастеров для `master.local` в `/etc/hosts` (или реализовать DNS сервер).
   - *Вариант*: Агент получает список всех мастеров при `join` и периодически обновляет его.
4. **VPN через публичные VPS**:
   - Агенты могут использовать любой из публичных IP мастеров как эндпоинт WireGuard.

## Ожидаемый результат
1. Можно запустить `r4a-server init` на первом мастере.
2. Можно запустить `r4a-server join-master --master <first-master-ip>` на втором мастере.
3. Оба мастера имеют одинаковые данные (список нод, репозитории).
4. Агент, подключенный к кластеру, видит `master.local` и запросы к нему балансируются (например, через Round-robin в DNS или hosts).

## План реализации
1. Исследовать `openraft` и спроектировать структуру `r4a-store`.
2. Реализовать `r4a-store` с базовой репликацией "списка нод".
3. Изменить `r4a-server` для поддержки нескольких мастеров.
4. Обновить логику резолвинга `master.local` в `r4a-agent`.
5. Протестировать деплой на `asus` и `home` (оба как мастера для теста).
